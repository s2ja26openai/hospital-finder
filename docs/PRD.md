# 요구사항 정의서: 내 주변 병원 찾기

---

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | 내 주변 병원 찾기 |
| 목적 | 사용자가 증상 또는 진료과 기준으로 현재 위치 기반 최적 병원을 추천받는 서비스 |
| 주요 사용자 | 갑작스러운 증상이 생겨 적합한 병원을 빠르게 찾고 싶은 일반 사용자 |
| 개발 언어 | Python (FastAPI) |
| 배포 환경 | Render |

### 1-0. 문제 배경 및 시장 니즈

**사용자 페인 포인트:**
- 갑작스러운 증상 발생 시 어느 진료과를 가야 할지 모르는 경우가 빈번함
- 네이버·카카오 지도로 병원 검색 시 진료과 필터링이 없어 일일이 확인해야 함
- 현재 운영 중인 병원인지 확인하려면 전화나 홈페이지를 별도 확인해야 함

**시장 규모 및 근거:**
- 국내 의원·병원 수: 약 73,000개 이상 (건강보험심사평가원, 2023)
- 스마트폰 기반 병원 검색 경험률: 국내 성인의 68% (한국인터넷진흥원, 2022 인터넷이용실태조사)
- 네이버 지도 월간 활성 사용자(MAU): 약 3,000만 명 — 병원 검색이 상위 카테고리
- 증상 기반 진료과 추천 서비스 공백: 국내 주요 플랫폼(네이버, 카카오) 모두 미제공

**타겟 사용자 시나리오:**
1. 갑자기 귀가 아픈데 이비인후과인지 신경과인지 모르는 직장인
2. 아이가 복통을 호소하는데 가까운 소아과가 지금 영업 중인지 확인하고 싶은 부모
3. 이사 후 새 동네에서 신뢰할 수 있는 내과를 찾는 1인 가구

## 1-1. 성공 기준 및 KPI

| 지표 | 목표 |
|------|------|
| 병원 검색 응답 속도 | 3초 이내 |
| 위치 기반 병원 조회 성공률 | 95% 이상 (Kakao API 기준) |
| 챗봇 진료과 추천 정확도 | 테스트 케이스 20개 기준 80% 이상 |
| 리뷰 수집 성공률 | 네이버 Place 검색 성공 시 50개 이상 수집 |
| 반응형 지원 | 모바일(375px) / 태블릿(768px) / PC(1440px) 전 화면 정상 렌더링 |

## 1-2. 경쟁 서비스 분석 및 차별점

| 서비스 | 특징 | 한계 |
|--------|------|------|
| 네이버 지도 병원 검색 | 방대한 병원 정보, 리뷰 | 증상 기반 진료과 추천 없음 |
| 닥터나우 | 비대면 진료 특화 | 오프라인 병원 탐색 목적 아님 |
| 당신의 의사 | 증상 입력 기능 | 위치 기반 필터링 약함 |
| 카카오맵 병원 | 위치 기반 탐색 | AI 증상 분석 없음, 리뷰 요약 없음 |

**본 서비스 차별점:**
1. **증상 → 진료과 자동 추천** (Claude API): 사용자가 의학 지식 없이도 적합한 과를 찾을 수 있음
2. **리뷰 기반 평점 정렬**: 네이버 리뷰를 수집·분석하여 신뢰도 높은 병원을 우선 추천
3. **실시간 운영 상태 표시**: 현재 진료 중/진료 시작 예정/오늘 휴무를 실시간으로 구분 표시
4. **단일 서비스 통합**: 증상 분석 + 위치 기반 탐색 + 리뷰 평가를 하나의 플로우로 제공

## 2. 핵심 기능 목록 (Feature List)

### F-01. 병원 탐색 진입 방식 (2가지)

| ID | 기능 | 설명 |
|----|------|------|
| F-01-A | 증상 기반 탐색 | 챗봇에 증상을 입력 → 진료과 추천 → 병원 탐색 |
| F-01-B | 진료과 직접 선택 | 드롭다운/버튼으로 진료과 직접 선택 |

---

### F-02. 증상 기반 진료과 추천 (챗봇)

| ID | 요구사항 |
|----|---------|
| F-02-01 | 사용자가 자연어로 증상을 입력할 수 있어야 한다 |
| F-02-02 | 입력된 증상을 분석하여 가능성 있는 진료과 후보 목록(복수)을 추천해야 한다 |
| F-02-03 | 각 진료과 후보에 대해 해당 추천 근거(질병 가능성)를 함께 표시해야 한다 |
| F-02-04 | 사용자는 추천된 진료과 목록 중 하나를 선택하여 병원 탐색으로 진행한다. 질병명은 선택 단위가 아닌 근거 정보로만 표시한다 |
| F-02-05 | 챗봇은 진료과 추천의 정확도를 높이기 위해 추가 증상을 되묻는 멀티턴 대화를 지원할 수 있어야 한다 (선택) |

---

### F-03. 위치 설정

| ID | 요구사항 |
|----|---------|
| F-03-01 | 기본값은 사용자의 현재 위치(GPS)를 사용한다 |
| F-03-02 | 사용자가 직접 주소 또는 지명을 입력하여 기준 위치를 변경할 수 있어야 한다 |
| F-03-03 | GPS 권한 거부 시 주소 입력으로 fallback 처리해야 한다 |

---

### F-04. 병원 검색 및 필터링

| ID | 요구사항 |
|----|---------|
| F-04-01 | 기준 위치 반경 내 병원 목록을 조회해야 한다. 반경 기본값은 500m이며, 사용자가 500m / 1km / 5km / 10km 중 선택할 수 있어야 한다 |
| F-04-02 | 선택한 진료과 또는 해당 전문의가 있는 병원만 필터링해야 한다 |
| F-04-03 | 오늘 날짜 기준 휴무일이 아닌 병원을 우선 표시해야 한다. 휴무일이 아니나 현재 시간 기준 미운영 중인 병원은 다음 진료 시작 시간을 `00:00 진료시작` 형태로 표시해야 한다. 휴무일인 병원은 목록 하단에 표시한다 |

**병원 표시 우선순위:**

| 순위 | 상태 | 표시 예시 |
|------|------|-----------|
| 1 | 휴무 아님 + 현재 운영 중 | `진료 중 · 21:00 마감` |
| 2 | 휴무 아님 + 현재 미운영 | `09:00 진료시작` |
| 3 | 오늘 휴무 | `오늘 휴무` |

---

### F-05. 정렬 기준

| ID | 요구사항 |
|----|---------|
| F-05-00 | 정렬 기준의 기본값은 평점순이며, 사용자가 거리순으로 변경할 수 있어야 한다 |

#### F-05-B. 평점순 (네이버 리뷰 분석)

| ID | 요구사항 |
|----|---------|
| F-05-B-01 | 네이버 지도 리뷰를 최신순으로 최대 50개까지 수집하여 분석해야 한다 |
| F-05-B-02 | 리뷰를 AI로 요약하여 긍정/부정 포인트를 간략히 제공해야 한다 |
| F-05-B-03 | 요약 기반 점수는 `긍정 포인트 개수 - 부정 포인트 개수`로 산출하며, 점수가 높은 순으로 정렬해야 한다 |
| F-05-B-04 | 리뷰 수가 10개 미만인 경우 신뢰도 낮음 표시 또는 정렬 제외 처리해야 한다 |

---

### F-06. 병원 목록 및 상세 정보

| ID | 요구사항 |
|----|---------|
| F-06-01 | 병원명, 진료과/전문의, 거리, 운영 여부, 리뷰 요약을 카드 형태로 표시해야 한다 |
| F-06-02 | 지도 위에 병원 위치 핀을 함께 표시해야 한다 |
| F-06-03 | 사용자가 병원 카드를 선택하면 상세 정보(전화번호, 주소, 전체 운영시간)를 확인할 수 있어야 한다 |

## 3. 데이터 요구사항

| 데이터 | 출처 |
|--------|------|
| 병원 정보 (위치, 진료과, 운영시간) | 건강보험심사평가원 API, 공공데이터포털 |
| 네이버 리뷰 | 네이버 지도 크롤링 또는 API |
| 증상 → 진료과 추론 | Claude API |

## 4. 비기능 요구사항

| 항목 | 요구사항 |
|------|---------|
| 개인정보 | 위치 정보는 탐색 목적 외 저장 금지 |

## 5. 사용자 플로우 요약

```
[시작]
  ├─ 증상 입력 (챗봇)
  │     └─ 진료과 후보 추천 (질병은 근거로 표시) → 진료과 선택
  └─ 진료과 직접 선택
        │
        ▼
  위치 확인 (GPS or 주소 입력)
        │
        ▼
  병원 목록 조회 (진료과 필터 + 휴무일/운영시간 필터)
  반경 선택: [500m (기본)] / 1km / 5km / 10km
        │
        ▼
  정렬 선택: [평점순 (기본)] or [거리순]
        │
        ▼
  병원 카드 목록 + 지도 표시
        │
        ▼
  병원 선택 → 상세 정보 확인
```
